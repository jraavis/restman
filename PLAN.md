# Restman — Development Plan

Privacy-first Tauri v2 (Rust + React) REST client. Frontend talks to the backend only over Tauri IPC; Rust owns all networking, file I/O, and crypto. Secrets (tokens, passwords, API keys) never cross IPC in plaintext and are never persisted to SQLite in plaintext — see the mask-on-write contract in `src-tauri/src/secrets.rs` and `SECRET_MASK` in `src/lib/types.ts`. Storage is SQLite only.

Repo: https://github.com/jraavis/restman (public, default branch `main`).

**17b — WebSocket ephemeral streaming client ✅ complete**
- **Transport parity (the load-bearing decision).** A naive `tokio_tungstenite::connect_async` would open its own socket and silently ignore the per-workspace proxy / client-cert / default-header transport settings shipped in the Interlude — fatal behind a corporate proxy. Instead the handshake runs *through* the same `engine::http::build_client` reqwest client SSE/HTTP use: `engine::ws::connect` sends the upgrade GET (`ws(s)://`→`http(s)://`, generated `Sec-WebSocket-Key`, `Upgrade`/`Connection`/`Version: 13` headers), requires `101`, validates `Sec-WebSocket-Accept` via `tungstenite::handshake::derive_accept_key`, then `resp.upgrade()` → `WebSocketStream::from_raw_socket(_, Role::Client, None)`. reqwest owns the connection and TLS, so proxy/mTLS/default-headers all carry over and tokio-tungstenite needs **no TLS feature** (it only frames an already-secured byte stream) — sidesteps the native-tls/rustls backend-mismatch question entirely. This replicates `reqwest-websocket`'s mechanism without taking the dep (it needs reqwest 0.13; we're on 0.12). Verified the API shapes against the vendored crate sources before committing: `reqwest::Response::upgrade()`/`Upgraded: AsyncRead+AsyncWrite+Unpin` exist; reqwest 0.12 has **no `upgrade` cargo feature** (the method is ungated — do not add one); `from_raw_socket` and `derive_accept_key` are reachable with only tokio-tungstenite's `handshake` feature. One gap found after the initial pass: `build_client`'s default `Client` lets ALPN pick HTTP/2 on a `wss://` connection, and h2 has no `Upgrade:` mechanism (RFC 7540 §8.1.2.2 strips it) — the 101 response would never arrive against any h2-capable server. Fixed by splitting `build_client` into a shared `build_client_inner` and a new `build_ws_client` that pins `.http1_only()`; only the WS handshake client forces HTTP/1.1, ordinary HTTP/SSE traffic is untouched.
- **Backend**: `model::streaming::WsEvent` (`Open`/`Message{binary,data}`/`Closed{code,reason}`/`Error{message}`, tagged `type`) + `WsOutbound{binary,data}` + two pure converters — `message_to_event` (Text→text, Binary→base64, Close→code/reason, **Ping/Pong suppressed** as transcript noise since tungstenite auto-replies to Ping on the next write) and `outbound_to_message` (base64-decodes binary, errors on bad base64). `engine::ws` (handshake + framing; the `Sec-WebSocket-Accept` derivation is KAT-tested against the RFC 6455 vector, the one piece verifiable offline). `commands::streaming`: `ws_connect` (resolves transport, builds client, spawns a `select!` driver splitting inbound→`Channel<WsEvent>` and an `mpsc` outbound→sink), `ws_send` (decodes via `outbound_to_message` so bad base64 surfaces synchronously to the caller, not swallowed in the loop), and a **single generic `stream_disconnect`** serving SSE+WS+future gRPC (abort-and-drop needs no protocol knowledge).
- **AppState shape change**: `streams` went from `HashMap<String, JoinHandle>` to `HashMap<String, StreamHandle { task, sender: Option<UnboundedSender<WsMessage>> }>` — one unified map (not two parallel maps) so there's no sender/handle desync class; SSE entries carry `sender: None`. The `sender` holds tungstenite `Message` (converted in `ws_send`) rather than `WsOutbound`, which is what lets `ws_send` return base64 errors synchronously.
