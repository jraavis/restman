# Restman — Development Plan

Privacy-first Tauri v2 (Rust + React) REST client. Frontend talks to the backend only over Tauri IPC; Rust owns all networking, file I/O, and crypto. Secrets (tokens, passwords, API keys) never cross IPC in plaintext and are never persisted to SQLite in plaintext — see the mask-on-write contract in `src-tauri/src/secrets.rs` and `SECRET_MASK` in `src/lib/types.ts`. Storage is SQLite only.

Repo: https://github.com/jraavis/restman (public, default branch `main`).

**17b — WebSocket ephemeral streaming client ✅ complete**
- **Transport parity (the load-bearing decision).** A naive `tokio_tungstenite::connect_async` would open its own socket and silently ignore the per-workspace proxy / client-cert / default-header transport settings shipped in the Interlude — fatal behind a corporate proxy. Instead the handshake runs *through* the same `engine::http::build_client` reqwest client SSE/HTTP use: `engine::ws::connect` sends the upgrade GET (`ws(s)://`→`http(s)://`, generated `Sec-WebSocket-Key`, `Upgrade`/`Connection`/`Version: 13` headers), requires `101`, validates `Sec-WebSocket-Accept` via `tungstenite::handshake::derive_accept_key`, then `resp.upgrade()` → `WebSocketStream::from_raw_socket(_, Role::Client, None)`. reqwest owns the connection and TLS, so proxy/mTLS/default-headers all carry over and tokio-tungstenite needs **no TLS feature** (it only frames an already-secured byte stream) — sidesteps the native-tls/rustls backend-mismatch question entirely. This replicates `reqwest-websocket`'s mechanism without taking the dep (it needs reqwest 0.13; we're on 0.12). Verified the API shapes against the vendored crate sources before committing: `reqwest::Response::upgrade()`/`Upgraded: AsyncRead+AsyncWrite+Unpin` exist; reqwest 0.12 has **no `upgrade` cargo feature** (the method is ungated — do not add one); `from_raw_socket` and `derive_accept_key` are reachable with only tokio-tungstenite's `handshake` feature. ### Phase 6, task 5 of 5 — Plugin system for custom codegen / custom import/export formats ✅ complete

Scope was confirmed with the user via two explicit questions before starting (not defaulted, per this phase's own precedent — see task 4's intro): **both** codegen and import/export plugins in this one pass (not a subset), **and** storage/discovery/management UI in scope now (not deferred to a follow-up spike) — a bigger, full-feature scope than task 4's protocol-by-protocol staging.

**Design.** The central question was what runs a plugin. This codebase already had a proven sandboxed JS execution environment from Phase 4 (`rquickjs`/QuickJS — fresh `Runtime`+`Context` per run, no filesystem/network access, 8s timeout, 512KB stack) built for pre/post-request scripts. Reused that sandbox primitive rather than introducing a new execution mechanism (no WASM runtime, no external process, no new trust boundary to audit) — but a plugin call is a different shape than a script run: scripts are side-effecting (mutate shared state via an injected `pm` object), a plugin is a pure function call (JSON in, typed value out). The new `plugins::runtime` module is that distinct contract: evaluate plugin source, look up a named top-level function, call it with JSON-marshaled args, marshal the return value back (`call_returning_string` for a required JS-string return; `call_returning_json<T>` round-tripping through `JSON.stringify` + `serde_json::from_str` for structured returns). Secrets: only the codegen path needed new masking — `codegen::plugin::mask_auth_for_plugin` replaces every non-empty `AuthConfig` secret field with a placeholder before a request reaches a plugin (native generators are trusted Rust in this binary and already see resolved auth; plugins are untrusted third-party JS and never see a real secret). Import/export plugins needed none: `interop::collect`'s `ImportedNode` output already reads mask-on-write `auth_json` (this module's existing contract), and a `parse` plugin's output flows through the same `apply_import` pipeline as every native parser, routing any auth through `crate::auth::persist` before it touches the DB.

**Storage**: workspace-scoped `plugins` table (migration v6) — `id`/`workspace_id`/`name`/`kind` (`codegen`/`import`/`export`)/`language_label` (display name)/`source` (JS text)/`enabled`/timestamps. Workspace-scoped rather than app-global to match every other piece of user data in this codebase (collections, environments, workspace settings), not because plugins are conceptually tied to one workspace's *data* — same precedent as the Interlude's per-workspace transport settings.

**Backend dispatch.** Both `commands::codegen::generate_code` and `commands::interop::{preview_import, export_collection}` needed a mutually-exclusive native-vs-plugin selector. Codegen uses a tagged `CodegenTarget::{Native{language}, Plugin{plugin_id}}` enum (a two-`Option` pair was tried first but pushed `generate_code` to 8 parameters, one over clippy's `too_many_arguments` threshold — the enum fixed both the type-safety and the lint at once). Import/export kept two plain `Option`s (`format`/`plugin_id`) since those commands had headroom under the parameter limit. `codegen::plugin::generate`/`interop::plugin::{parse, export}` are the per-domain glue: JSON-serialize the relevant native type, call into `plugins::runtime`, deserialize/return. ## Phase 8 — Data, Backup, Distribution ✅ complete (updater wired with an ephemeral dev key, unpublished)

- **File-based `.restman/` sync** (`src-tauri/src/sync.rs`, `commands/sync.rs`): migration v8 adds `sync_folder_path`/`sync_mode` (`off`/`manual`/`live`)/`sync_format` (`json`/`yaml`) to `workspace_settings`. Reuses the existing import/export IR instead of inventing a new format — `interop::{ImportedNode, collect, apply_import}` for collections (already `Serialize`/`Deserialize`, so the native JSON/YAML file *is* that struct) and `interop::environment::{export_environment, parse, apply_environment_import}` for environments; the only new code is folder layout, a local JSON↔YAML bridge (`saphyr`, reusing the crate already pinned for OpenAPI YAML import — no new dependency), and file naming (`slugify`). History still ships in full inside the backup ZIP below, where "protected, not redacted" is the right trade-off. `export_to_folder` (DB → files) is the only thing ever auto-triggered (`SyncMode::Live`); `import_from_folder` (files → DB) is always an explicit user action — round-tripping external file edits back into the DB without a conflict-resolution engine (out of scope this pass) risks silently clobbering concurrent DB changes.
